# 需求文档

## 1. 项目现状与核心目标

当前正在将 upstream/main 的变更手动合并到本地分支,本轮对应版本为 0.6.65.

这次冲突不是单个文件的小修小补,而是两边同时改到了同一条会话主链路. 上游本轮的核心价值主要有 3 类:

- 会话内核重构,把原本集中在 `useConversation.ts` 的主流程拆到 `core/*`.
- 子代理流式处理增强,补齐子代理消息在流式过程中的写入与持久化能力.
- 新增 `/gitline` 相关命令与面板链路.

本地必须继续保留的核心能力也很明确:

- 主代理体系与主代理相关配置入口.
- 提问工具显示与 ESC 占用修复相关行为,不得在合并后回退.
- 主代理/子代理的 system prompt 与角色定义隔离语义.
- 可编辑文件类型限制.
- Response API 对图片拆分为独立 user 消息的本地兼容增强.
- 已明确废弃的旧模式体系不回流,尤其是 `role`, `planMode`, `vulnerabilityHuntingMode`, `--yolo` 等旧入口.

本次文档目标是明确 0.6.65 这一轮手动合并的最终口径,让相关文件在同一套规则下完成合并,避免一个文件按上游思路,另一个文件按本地旧思路,最终造成行为不一致.

### 1.1 术语说明

- `core/*` 拆分: 可以理解为把一个过大的总开关,拆成几个职责明确的小开关. 终态是主流程更清楚,不是把旧的大文件原样塞回来.
- 子代理流式处理: 可以理解为子代理工作时,内容是一边产生一边显示,而不是最后一次性出现.
- 持久化与回放: 可以理解为当用户退出再回来时,仍能看到之前的关键过程和结果,而不是只在当时短暂显示过.
- 最小完整链路: 可以理解为一个功能从入口到结果必须完整打通,不允许只有按钮,没有后续动作,也不允许只有面板,没有入口.

## 2. 范围与边界

**功能点简述**:

- [ ] 吸收 upstream 的会话内核重构,以 `core/*` 拆分后的结构为主干.
- [ ] 吸收 upstream 的子代理流式处理能力,包括消息写入,回放相关元信息与工具调用协同.
- [ ] 保留本地主代理体系,提问工具修复,文件类型限制等既有能力.
- [ ] 明确 `/role` 不合入,不得以任何形式回流到命令,面板,文案,状态或类型定义中.
- [ ] 保留 `/gitline`,但只保留它所需的最小完整链路,不顺带恢复其他无关上游命令体系.
- [ ] 明确冲突涉及的类型,命令面板,输入面板,i18n,system prompt 等连带文件必须同步对齐.
- [ ] 合并完成后,必须按本地特殊修改清单做一次全量复查,确认没有丢失,也没有破坏上游新增逻辑.

**排除项**:

- 不恢复 `role` 旧体系.
- 不恢复 `planMode`, `vulnerabilityHuntingMode`, `--yolo` 等已废弃模式入口.
- 不在本轮需求文档中记录合并过程日志.
- 不为了合并顺手引入与本轮冲突无关的新重构.
- 不把 `/gitline` 扩展成超出上游原始语义的本地新功能.

## 3. 合并后的终态要求

### 3.1 会话主链路

- `useConversation.ts` 的终态应以上游 `core/*` 拆分架构为主,主文件负责编排,而不是重新塞回大量内联逻辑.
- 本地必须保留的能力,应接到新的分层边界中,而不是通过回退到旧结构来实现.
- 会话初始化,用户消息写入,流式轮次处理,工具调用轮次处理之间的职责边界必须清晰稳定.

### 3.2 子代理流式处理

- 必须保留上游对子代理流式消息的支持.
- 子代理相关消息在 UI 展示,持久化字段,回放语义上必须互相匹配,不能只改一半.
- 若类型中新增子代理流式相关字段,不得覆盖或破坏本地已有的特殊 user 消息标记语义,应以能力并存为准.

### 3.3 ChatMessage 字段契约

- `specialUserMessage` 继续表示本地动态注入的特殊 user 上下文消息.
- `subAgentContent` 继续表示子代理流式过程中的可回放内容标记.
- `subAgent` 元信息继续用于标识子代理身份与完成状态.
- 上述字段必须允许并存,不能通过“保留一个,删除一个”的方式解决冲突.
- 需要保存到会话历史中的字段,在持久化时必须保留; 仅用于过滤发给模型的内部标记,不得误删用户需要回看的记录.

### 3.4 命令与面板链路

- `/gitline` 需要保留,但只保留它的最小完整闭环:
  - 命令注册.
  - 命令面板展示.
  - 输入框与键盘交互.
  - GitLinePickerPanel 面板显示与选择结果写回输入框.
  - 所需 i18n 文案.
- `/gitline` 的交互终态必须明确:
  - 用户可以通过输入 `/gitline` 或在命令面板中选择它来触发.
  - 打开面板后,用户可以正常取消返回,不能出现进得去退不出的状态.
  - 选择结果应写回当前聊天输入框,而不是丢到其他位置.
- `/role` 不得保留:
  - 不出现在命令注册中.
  - 不出现在命令面板中.
  - 不出现在输入处理逻辑中.
  - 不出现在 i18n 文案与类型定义中.
- `/yolo` 与旧模式入口同样不得借本轮合并回流到命令层,面板层或文案层.

### 3.5 system prompt 与本地特殊架构

- 本地主代理/子代理 system prompt 与角色定义隔离语义必须继续成立.
- 已明确废弃的旧模式字段,不得借由上游重构重新回流到会话参数,状态传递,提示词组装或界面展示中.
- `systemPrompt.ts` 如存在删除与修改冲突,最终结果必须服从当前本地主代理体系,不能借恢复旧文件而带回旧模式耦合.

### 3.6 提问工具与 ESC 行为

- 本地已修复的提问工具显示与 ESC 占用问题,在本轮合并后必须继续成立.
- 不能因为上游会话主链路重构或输入逻辑调整,导致提问选项再次只显示一半,或回答后 ESC 仍被错误占用.
- 相关输入处理,面板显示与会话中断逻辑需要按同一口径校验.

### 3.7 本地特殊能力全量保留

以下能力在本轮合并后必须逐项确认仍然成立:

- 主代理系统与主代理切换入口.
- 子代理配置体系与 TOML 配置驱动方式.
- 编辑文件后缀限制能力.
- Response API 图片拆分兼容路径.
- todo 树形展示与 `/clear`, `/resume` 后的刷新语义.
- bash 优先使用用户当前 shell 的执行策略.

## 5. 举例覆盖需求和边缘情况

- **例 1: 会话主链路合并**

  - 场景: 上游把主流程拆到 `core/*`,本地仍有主代理和提问工具相关定制.
  - 期望: 最终结构采用拆分后的 `core/*` 架构,同时本地定制能力仍正常工作.
  - 不应出现: 为了保留本地能力,整个 `useConversation.ts` 回退成旧版大文件实现.

- **例 2: 子代理流式输出**

  - 场景: 主代理调用子代理,子代理在执行中持续输出消息.
  - 期望: UI 中能持续看到子代理流式内容,相关消息持久化与回放字段也完整可用.
  - 不应出现: 界面能看到内容,但历史回放丢失,或类型字段不完整导致后续处理异常.

- **例 3: `/gitline` 最小保留**

  - 场景: 用户输入 `/gitline`,或者在命令面板中选中 `/gitline`.
  - 期望: 命令面板和输入框都能正确打开 gitline 面板,用户可以取消返回,也可以选择内容并写回当前输入框.
  - 不应出现: 命令名出现在面板里,但没有面板实现,或键盘交互接不起来,或选完后内容没有回到输入框.

- **例 4: `/role` 必须彻底移除**

  - 场景: 合并上游后检查命令体系.
  - 期望: 用户看不到 `/role`,类型和文案中也没有残留.
  - 不应出现: 命令本身删了,但 i18n 里还有 `role` 文案,或输入处理里还有分支残留.

- **例 5: specialUserMessage 与子代理字段并存**

  - 场景: `ChatMessage` 同时需要支持本地特殊 user 消息标记,以及上游子代理流式持久化元信息.
  - 期望: 两类能力都能表达,互不覆盖.
  - 不应出现: 为了合入上游字段,把本地 `specialUserMessage` 语义删掉.

- **例 6: 提问工具与 ESC**

  - 场景: 用户触发提问工具后误按 ESC,随后完成回答.
  - 期望: 静态显示区能看到问题与建议选项,回答结束后 ESC 占用被正确释放.
  - 不应出现: 只能看到问题看不到选项,或回答结束后 ESC 仍无法中断会话.

- **例 7: system prompt 冲突**

  - 场景: 上游修改 `systemPrompt.ts`,本地则已移除旧模式并建立主代理/子代理提示词体系.
  - 期望: 最终提示词体系继续服从当前本地架构,不带回旧模式入口.
  - 不应出现: 合并后又重新出现 `role` 旧入口,或旧模式字段重新参与提示词组装.

- **例 8: 合并后全量复查**
  - 场景: 冲突文件都已解决并 build 通过.
  - 期望: 仍需按本地特殊修改清单逐项复查,确认没有“表面能编译,实际本地能力已悄悄丢失”的情况.
  - 不应出现: 只看冲突文件通过就结束,遗漏主代理,文件类型限制,Response API 图片兼容等关键本地能力.

## 6. 验收口径

- 必须能完成 build,不能仍保留任何 merge conflict 标记.
- 必须全局确认 `role`, `planMode`, `vulnerabilityHuntingMode`, `--yolo` 等本轮明确排除项没有以命令,类型,文案,状态或提示词路径形式残留.
- 必须手动验证 `/gitline` 的打开,取消,选择回填 3 个动作都成立.
- 必须手动验证提问工具的静态显示区与 ESC 释放行为没有回退.
- 必须按本地特殊修改清单逐项复查主代理体系,文件类型限制,Response API 图片兼容,todo 树形展示与 shell 选择策略.
