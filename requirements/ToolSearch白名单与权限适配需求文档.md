# 需求文档

## 1. 项目现状与核心目标

当前项目中的 Tool Search 系统已经接入主代理主链路. 其现状是: 在对模型发起请求时,初始只直接提供 `tool_search` 工具,其他工具只有在模型先调用 `tool_search` 搜索到后,才会在后续轮次中变为可见且可调用.

这套机制已经具备“渐进暴露工具”的能力,但在本地使用上存在 3 个明显问题:

1. 某些高频常用工具仍要先绕一次 `tool_search`,使用成本偏高.
2. Tool Search 目前需要进一步和主代理/子代理已有的工具权限配置完全对齐.
3. 子代理的 3 个协作工具 `send_message_to_agent`, `query_agents_status`, `spawn_sub_agent` 目前仍存在特殊处理路径,尚未完全纳入统一的工具权限管理.

本需求的核心目标是:

- 在不破坏现有 Tool Search 渐进暴露架构的前提下,增加“代码内白名单”机制.
- 让 Tool Search 与主代理/子代理的工具权限配置完全一致.
- 让子代理 3 个协作工具与普通工具一样,直接受现有工具权限名单控制.
- 最终收敛出一条清晰的统一规则: “先算代理真正允许看到的工具集合,再决定哪些初始直接暴露,哪些需要通过 `tool_search` 搜索后暴露.”

## 2. 范围与边界

**功能点简述**:

- [ ] 为 Tool Search 系统新增“代码内白名单”能力.
- [ ] 白名单中的工具可跳过 Tool Search 渐进暴露,在初始请求中直接提供.
- [ ] 白名单工具仍必须经过主代理/子代理的工具权限过滤.
- [ ] 没有权限的工具,即使在白名单中,也不能出现在代理工具列表中.
- [ ] 没有权限的工具,也不能在 Tool Search 搜索结果中出现.
- [ ] 子代理的 3 个协作工具纳入现有 `availableTools`/工具权限名单统一管理.
- [ ] 子代理未配置某个协作工具时,该工具既不可见,也不可搜索,也不可调用.

**排除项**:

- 不新增用户可编辑的白名单配置 UI.
- 白名单仅写在代码中,不暴露给用户配置.
- 不新增一套独立的“协作工具权限组”配置字段.
- 不改变 Tool Search 的基本定位,即它仍然是“按需发现工具”的机制.
- 不改变主代理和子代理原有工具权限字段的基础语义,只在现有语义上补充适配规则.
- 不要求在本需求中重做普通工具的权限配置 UI.

## 3. 需求定义

### 3.1 代码内白名单的基本定义

本需求中的代码内白名单分为两层:

1. 全局工具白名单.
2. 主代理跳过搜索子代理白名单.

其中:

- 全局工具白名单对主代理和子代理都生效.
- 主代理跳过搜索子代理白名单只对主代理生效,子代理侧不使用这份白名单.
- 这两份白名单在代码中的默认初始值都应为空.
- 初始实现阶段不预置任何默认条目,由用户后续手动填写代码中的白名单内容.

白名单的定义形式都是“具体工具全名白名单”,不是工具分类白名单.

这里的“具体工具全名”指“服务名 + 工具名”的完整标识,而不是只写工具短名.

白名单匹配规则必须与现有工具权限筛选口径保持一致,采用严格精确匹配,不允许前缀匹配,模糊匹配或包含匹配.

也就是说:

- 白名单中写什么完整标识,就只匹配那个完整工具.
- 不能因为写了某个前缀,就把一组工具都放进白名单.
- 不能因为工具名部分相似,就视为命中白名单.

全局工具白名单中的条目必须写完整标识,例如:

- `todo.todo-get`
- `filesystem.filesystem-read`
- `functions.tool_search`

主代理跳过搜索子代理白名单中的条目也必须写完整标识.若子代理工具在运行时呈现为 `subagent-agent_xxx` 这一完整工具名,则白名单中也必须写该完整工具名,不能只写子代理 id.

两层白名单的职责只有一个: 决定某个工具是否“在首轮请求中直接暴露”.
它们都不负责绕过权限检查.
它们都不负责绕过权限检查.

### 3.2 白名单与权限过滤的优先级

本需求明确以下优先级:

1. 先计算“某个代理实际允许看到的工具集合”.
2. 再从这个集合中,挑出属于白名单的工具,作为“初始直接暴露工具”.
3. 对主代理来说,如果某个 `subagent-xxx` 同时命中“主代理跳过搜索子代理白名单”,则它也可作为主代理的初始直接暴露工具.
4. 不在任何适用白名单中的允许工具,仍走 Tool Search 渐进暴露流程.

换句话说:

- 白名单只是“跳过 Tool Search 渐进暴露”.
- 白名单不是“绕过权限过滤”.
- 主代理跳过搜索子代理白名单,本质上只是主代理场景下的一个额外初始暴露层.
- 白名单命中判断必须采用严格精确匹配,与现有权限筛选规则一致.

### 3.3 主代理的权限适配规则

对于主代理:

1. 主代理已有的工具权限配置仍然是最终权限来源.
2. 对主代理来说,每个子代理本质上也是一个工具,其名称视为对应的 `subagent-xxx` 工具.
3. 因此,主代理的普通工具和子代理工具必须使用同一套可见性规则.
4. 只有被主代理权限允许的工具,才有资格进入:
   - 初始直接暴露集合.
   - Tool Search registry.
   - Tool Search 搜索结果.
5. 某工具若未被主代理权限允许:
   - 即使在全局工具白名单中,也不能在初始请求中出现.
   - 即使在主代理跳过搜索子代理白名单中,也不能在初始请求中出现.
   - 也不能被 `tool_search` 搜索到.
   - 也不能在上下文工具描述中出现.
6. 某个子代理若未被主代理配置为可调用,则对应的 `subagent-xxx` 工具也必须满足上述同样规则:
   - 不能初始直接暴露.
   - 不能被 Tool Search 搜索到.
   - 不能被调用.
7. 主代理跳过搜索子代理白名单的作用仅限于:
   - 对“主代理已经有权限调用的 `subagent-xxx` 工具”,允许其跳过 Tool Search 渐进暴露.
   - 不改变其权限来源.

### 3.4 子代理的权限适配规则

对于子代理:

1. 子代理已有的工具权限配置仍然是最终权限来源.
2. 子代理只使用“全局工具白名单”,不使用“主代理跳过搜索子代理白名单”.
3. 子代理初始直接暴露工具和可搜索工具,也必须先经过其自身工具权限过滤.
4. 某工具若未被子代理权限允许:
   - 即使在全局工具白名单中,也不能在初始请求中出现.
   - 也不能被 `tool_search` 搜索到.
   - 也不能在上下文工具描述中出现.

### 3.5 子代理 3 个协作工具纳入统一权限管理

以下 3 个工具在本需求中视为“与普通工具同级”的工具:

- `send_message_to_agent`
- `query_agents_status`
- `spawn_sub_agent`

本需求要求:

1. 这 3 个工具不再被视为“天然全局可见”.
2. 它们直接受子代理现有 `availableTools`/工具权限名单控制.
3. 某子代理若未配置其中某个工具,则该工具对该子代理必须满足以下 3 条:
   - 不出现在该子代理请求的 `tools` 中.
   - 不出现在 Tool Search 搜索结果中.
   - 即使模型构造了调用,执行阶段也必须拒绝.
4. 这 3 个工具与普通工具一样,也可以被白名单直接初始暴露,但前提仍然是该子代理本身拥有对应权限.

### 3.6 `spawn_sub_agent` 的额外约束继续保留

虽然 `spawn_sub_agent` 现在要纳入统一工具权限管理,但它仍然保留自己已有的额外约束:

1. 即使子代理拥有 `spawn_sub_agent` 工具权限,也仍然要继续受 `availableSubAgents` 白名单约束.
2. 即使子代理拥有 `spawn_sub_agent` 工具权限,也仍然要继续受运行时深度限制约束.
3. 因此,`spawn_sub_agent` 的最终可见与可执行条件必须同时满足:
   - 子代理工具权限允许.
   - 子代理 `availableSubAgents` 过滤后非空.
   - 运行时深度限制允许.
4. 任一条件不满足时:
   - 不应在可见工具中出现.
   - 不应被 Tool Search 搜索到.
   - 执行时仍需兜底拒绝.

## 4. 统一规则

### 4.1 工具可见性统一规则

对任意代理(主代理或子代理),某个工具是否可见,必须按以下顺序判断:

1. 该工具是否属于该代理当前权限允许集合.
2. 若不属于,则该工具完全不可见.
3. 若属于,再判断它是否以“完整工具标识”精确命中代码内白名单.
4. 若精确命中白名单,则该工具初始直接暴露.
5. 若不命中白名单,则该工具只进入可搜索集合,等待 `tool_search` 发现后再暴露.

### 4.2 Tool Search 搜索范围统一规则

1. Tool Search registry 本身也必须基于“当前代理权限允许看到的工具集合”构建.
2. Tool Search 只能搜索“当前代理权限允许看到的工具”.
3. Tool Search 绝不能搜索到当前代理没有权限的工具.
4. Tool Search 也绝不能把没有权限的工具描述暴露给模型.
5. 白名单工具如果已被初始直接暴露,仍然可以被 Tool Search 搜索到,但不应因为搜索而改变其权限状态.

### 4.3 执行期兜底规则

无论是普通工具还是 3 个协作工具,执行时都必须保留兜底校验.

也就是说:

1. 可见性过滤只是第一层.
2. Tool Search 搜索过滤是第二层.
3. 真正执行时仍必须再次按当前代理权限校验.
4. 对子代理 `spawn_sub_agent` 还必须额外做 `availableSubAgents` 和深度门禁的兜底校验.

## 5. 举例覆盖需求和边缘情况

**例 1: 白名单工具且有权限**

- 场景: `filesystem-read` 在代码内白名单中,主代理也拥有该工具权限.
- 期望: 会话首轮请求中就直接带上 `filesystem-read`.
- 不应出现: 还必须先搜索一次 `tool_search` 才能使用.

**例 2: 白名单工具但无权限**

- 场景: `filesystem-read` 在代码内白名单中,但当前主代理没有该工具权限.
- 期望: 该工具完全不可见.
- 不应出现: 因为进了白名单,所以仍出现在 `tools` 或搜索结果里.

**例 3: 不在白名单但有权限**

- 场景: `ace-text_search` 不在代码内白名单中,但当前主代理拥有权限.
- 期望: 首轮请求中不直接暴露该工具,但 `tool_search` 可以搜索到它.
- 不应出现: 它既不初始暴露,也无法被搜索到.

**例 4: 主代理跳过搜索子代理白名单命中且有权限**

- 场景: `subagent-agent_reviewer` 被写进“主代理跳过搜索子代理白名单”,且当前主代理确实配置了可调用 `agent_reviewer`.
- 期望: 该 `subagent-agent_reviewer` 工具可以在主代理首轮请求中直接暴露,不必先经过 Tool Search.
- 不应出现: 明明是主代理常用子代理,但仍必须先用 `tool_search` 搜索一次才可调用.

**例 5: 主代理对白名单子代理工具无权限**

- 场景: `subagent-agent_reviewer` 被写进“主代理跳过搜索子代理白名单”,但当前主代理未把 `agent_reviewer` 配置为可调用子代理.
- 期望: `subagent-agent_reviewer` 既不能初始直接暴露,也不能被 Tool Search 搜索到,也不能被调用.
- 不应出现: 因为命中主代理跳过搜索子代理白名单,它仍然被主代理首轮直接拿到或可搜索到.

**例 6: 子代理普通工具无权限**

- 场景: 某子代理没有 `filesystem-read` 权限.
- 期望: 该工具既不在初始工具中出现,也不在 Tool Search 搜索结果中出现.
- 不应出现: 只是因为主代理有权限或它属于全局白名单,子代理也能看见它.

**例 7: 子代理协作工具无权限**

- 场景: 某子代理未配置 `send_message_to_agent`.
- 期望: 该工具既不可见,也不可搜索,也不可调用.
- 不应出现: 它仍然像以前一样被自动注入.

**例 8: 子代理拥有 `send_message_to_agent` 权限**

- 场景: 某子代理在现有 `availableTools` 中明确配置了 `send_message_to_agent`.
- 期望: 它可以按白名单/Tool Search 规则暴露和调用该工具.
- 不应出现: 因为现在改成统一权限模型,反而把它完全禁掉.

**例 9: 子代理拥有 `spawn_sub_agent` 权限,但没有可 spawn 目标**

- 场景: 子代理工具权限允许 `spawn_sub_agent`,但 `availableSubAgents` 为空.
- 期望: `spawn_sub_agent` 仍然不可见,也不可搜索.
- 不应出现: 仅凭工具权限允许,它就出现在工具列表里.

**例 10: 子代理拥有 `spawn_sub_agent` 权限且白名单非空,但超过深度限制**

- 场景: 子代理具备工具权限,也配置了可调用子代理,但运行时深度门禁不允许继续 spawn.
- 期望: `spawn_sub_agent` 不应出现在最终可用工具中.
- 不应出现: 它仍然被 Tool Search 搜索到并可调用.

**例 11: 白名单包含协作工具,但子代理无权限**

- 场景: 全局工具白名单包含 `query_agents_status`,但某子代理未配置该权限.
- 期望: 它仍然完全不可见.
- 不应出现: 因为命中白名单,它被直接初始暴露.

**例 12: Tool Search 搜索结果过滤**

- 场景: 某代理搜索 “agent” 相关工具.
- 期望: 搜索结果只返回它当前有权限看到的协作工具或子代理工具.
- 不应出现: 返回没有权限的 `send_message_to_agent`, `spawn_sub_agent`,或未被主代理允许的 `subagent-xxx`.
