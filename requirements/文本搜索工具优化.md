# 文本搜索工具优化需求文档

## 1. 调研发现

### 1.1 工具概述

项目中的文本搜索工具 `ace-text_search` 是一个三层回退架构的代码搜索工具,位于:

- **核心实现**: `source/mcp/aceCodeSearch.ts`
- **调用入口**: `source/utils/execution/mcpToolsManager.ts`

### 1.2 三层搜索策略(按优先级回退)

**优先级 1: git grep** (最快)

- **触发条件**: 当前目录是 Git 仓库 且 git 命令可用
- **命令**: `git grep --untracked -n --ignore-case [-E|--fixed-strings] pattern`
- **特点**:
  - 利用 Git 索引,速度极快
  - 支持正则表达式(`-E`)和字面量搜索(`--fixed-strings`)
  - 搜索包括未跟踪文件(`--untracked`)
  - 不区分大小写(`--ignore-case`)
- **超时控制**: 无显式超时(依赖 git 自身速度)

**优先级 2: system grep/ripgrep** (快速)

- **触发条件**: rg 或 grep 命令可用
- **命令**:
  - ripgrep: `rg -n -i --no-heading pattern --glob !exclude/`
  - grep: `grep -r -n -H -E -i --exclude-dir pattern .`
- **特点**:
  - ripgrep 优先于 grep(性能更好)
  - 自动排除常见目录: `node_modules`, `.git`, `dist`, `build`, `__pycache__`, `target`, `.next`, `.nuxt`, `coverage`
- **超时控制**: **15 秒超时** (line 813, 862-872)
  - 超时后发送 `SIGTERM` 终止进程
  - 抛出错误: `${grepCommand} search timed out after 15000ms`

**优先级 3: JavaScript 纯实现** (最慢但最可靠)

- **触发条件**: 前两种都失败时的保底方案
- **特点**:
  - 递归遍历文件系统
  - 逐文件读取内容并使用 `RegExp` 匹配
  - 支持正则和字面量搜索
  - 自动跳过二进制文件(基于扩展名)
  - 使用配置化的目录排除规则
- **超时控制**: 无显式超时(可能在大代码库中很慢)

### 1.3 关键发现

**性能优化点**:

1. **命令可用性缓存** (line 167-175): 使用 `commandAvailabilityCache` 缓存命令检测结果,避免重复 `which` 查询
2. **Git 仓库状态缓存** (line 180-196): 使用 `isGitRepoCache` 缓存 Git 仓库检测结果
3. **文件修改时间排序** (line 1178-1235): 搜索结果按文件修改时间排序,最近修改的文件优先(24 小时内)
4. **文件统计缓存** (line 76-77): 使用 `fileStatCache` 缓存文件 stat 信息,TTL 60 秒

**Windows 兼容性**:

- 所有路径都进行归一化处理: 将 `\` 转换为 `/`
- glob 模式展开支持花括号: `*.{ts,tsx}` → `["*.ts", "*.tsx"]`

**文件过滤**:

- 支持文件 glob 参数(如 `*.ts`, `src/**/*.py`)
- 二进制文件自动跳过(基于扩展名集合, O(1) 查找)
- 可配置的排除目录(从配置文件加载)

**错误处理**:

- 搜索无结果时返回空数组,不报错(exit code 1 视为正常)
- 统一解析 grep 输出格式: `filePath:lineNum:content`
- 抑制常见无害错误: "Permission denied", "grep:.\*: Is a directory"
- 子进程注册到 `processManager` 确保清理

**潜在问题**:

1. **JavaScript 回退方案无超时**: 在大代码库中可能导致长时间阻塞
2. **git grep 无超时**: 在超大仓库中可能很慢
3. **缓存过期策略**: 文件统计缓存 60 秒 TTL,可能导致修改检测延迟
4. **内存占用**: 大文件全文读取到内存,可能消耗大量内存

### 1.4 相关机制:长命令后台执行

项目中的 `terminal-execute` 工具支持将长命令移至后台执行:

**触发方式**: 用户按 `Ctrl+B` 将当前运行的命令移至后台

**实现机制** (source/mcp/bash.ts, source/hooks/execution/useBackgroundProcesses.ts):

1. **命令启动**:

   - 命令启动后注册到后台进程列表
   - 记录 PID, 命令, 状态('running'/'completed'/'failed'), 启动时间

2. **移至后台** (Ctrl+B):

   - 设置 `shouldMoveToBackground = true` 标志
   - 后台检查循环(100ms 间隔)检测到标志后:
     - 立即返回当前已收集的输出
     - 添加提示: `[Command moved to background, execution continues...]`
     - 命令继续在后台运行

3. **状态管理**:

   - 命令完成时更新状态为 'completed' 或 'failed'
   - 记录完成时间和退出码
   - 用户可在 `BackgroundProcessPanel` 中查看所有后台进程

4. **UI 交互**:
   - 按 `Ctrl+B` 切换显示后台进程面板
   - 显示进程 PID, 状态, 运行时长
   - 支持手动 kill 进程

**关键发现**:

- 命令完成后**不会自动**将结果插入到后续消息中
- 仅更新后台进程列表状态
- 用户需要主动查看 BackgroundProcessPanel 获取结果

**与文本搜索的关系**:

- 当前文本搜索工具(ace-text_search)**未集成**后台执行机制
- 所有搜索都是同步阻塞的
- 如果搜索超时,会直接抛出错误(仅 system grep/ripgrep 有 15 秒超时)

## 2. 改造方案细化(讨论稿)

### 2.1 搜索优先级(保持不变,明确为硬约束)

- 先检查并优先使用 `git grep`.
- `git grep` 不可用时,回退到 `rg` 或 `grep`(优先 `rg`).
- JavaScript 纯实现仅作为"前两层报错/不可用"时的兜底.
- 超时场景不触发 JavaScript 兜底,避免在超大项目中继续放大耗时.

### 2.2 双超时机制

- 第一档超时: `15s`.
  - 前台立即返回占位提示,不再阻塞当前工具轮次.
  - 同一搜索任务转入后台继续执行.
- 第二档超时: `5min`.
  - 作为 `rg/grep` 层的后台超时上限.
  - 到达 5 分钟仍无结果时直接结束并标记异常.
  - 此场景直接返回超时失败,不再进入 JavaScript 兜底.

### 2.3 后台完成后的插回策略

- 后台任务完成后,不直接改 UI 消息列表.
- 结果写入等待消息队列(pendingMessages 同链路).
- 在最新一条工具执行完毕后,按现有 pending 消费机制插回为一条 `user` 消息.

### 2.4 搜索结果数量参数(`maxResults`)规则

- 工具入参继续保留 `maxResults`.
- 默认值维持 `100`(与当前实现一致).
- 调用方可显式传入更小或更大值.
- 建议增加边界保护:
  - 最小值: `1`.
  - 最大值: `500`.
  - 非法值(空,负数,NaN)回退到默认 `100`.

### 2.5 占位提示文案(建议)

`本次搜索时间较长,已转为后台异步搜索.结果完成后会自动插回当前会话.建议缩小搜索范围后重试.`

### 2.6 失败提示文案(建议)

`后台搜索已超时(5分钟),本次任务可能异常终止.建议缩小搜索范围后重试.`
