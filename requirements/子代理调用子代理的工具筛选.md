# 需求文档: 子代理调用子代理的工具筛选

## 1. 项目现状与核心目标

Snow CLI 支持“子代理(SubAgent)”在运行中调用工具.其中有 3 个“子代理协作相关工具”,用于让子代理与其他子代理进行协作或拉起新的子代理:

- `send_message_to_agent`
- `query_agents_status`
- `spawn_sub_agent`

当前系统在为某个子代理构建模型请求时,会把上述工具加入该子代理的 `tools` 字段中(只要子代理服务启用且满足深度等运行条件).

本需求使用子代理配置字段 `availableSubAgents`,用于控制 `spawn_sub_agent` 是否可用,以及可 spawn 的目标范围.

**核心目标**:

- `send_message_to_agent` 和 `query_agents_status` 是自带协作工具,保持全局可见,不随 `availableSubAgents` 开关.
- 当子代理配置“没有任何可 spawn 的子代理”时(字段未配置/`undefined` 或 `[]` 或清理后为空),不要把 `spawn_sub_agent` 加入该子代理请求的 `tools` 列表中.
- 当子代理配置存在可 spawn 子代理时,仅在符合运行条件时加入 `spawn_sub_agent`,并且其可选 `agent_id` 仅允许 `effectiveAvailableSubAgents` 中的子代理(即对白名单做无效 id 清理 + 排除自身后的结果).

## 2. 范围与边界

实现时需遵守项目既有的注释规范(中文,言简意赅,只解释 why,避免开发日志式注释).

**功能点简述**:

- [ ] 子代理配置字段: `availableSubAgents?: string[]`.
- [ ] 子代理构建请求时,根据 `availableSubAgents` 决定是否注入 `spawn_sub_agent`(以及其 `agent_id` 可选范围).
- [ ] UI/保存逻辑: 保存时清理无效的子代理 id.

**排除项**:

- 不改变主代理(MainAgent)的工具筛选逻辑.
- 不新增其他类型的协作工具.
- 不改变子代理自身已有的普通工具权限机制(例如 filesystem/terminal 等),本需求只约束上述 3 个协作工具.

## 3. 术语与定义(面向非专业读者)

- **子代理**: Snow CLI 中一种可被拉起并并行运行的辅助代理.
- **可调用子代理**: 当前子代理“允许拉起/协作”的其他子代理集合.
- **白名单**: `availableSubAgents` 指定的子代理 id 列表.
- **tools 列表**: 一次对话请求中,系统明确告诉模型“你现在能用哪些工具”的清单.如果某个工具不在 tools 列表中,模型就无法调用它.

## 4. 规则定义(无歧义)

### 4.0 现状差异与本需求的变更点(实现对齐)

为避免实现者误解,这里明确当前实现与本需求的差异:

- 当前实现倾向于在子代理请求中始终提供 `send_message_to_agent` 和 `query_agents_status`.
- 当前实现对 `availableSubAgents` 的处理更偏向“可选的 spawn 白名单”: 当白名单为空(包含未配置/`[]`)时,倾向于表示“不限制可 spawn 的子代理”,而不是“禁用 spawn”.

**本需求要求改为**:

- `send_message_to_agent` 和 `query_agents_status` 保持全局可见,不随 `availableSubAgents` 开关.
- 当子代理“没有任何可 spawn 的子代理”(字段未配置/`undefined` 或 `[]` 或清理后为空)时,`spawn_sub_agent` 不出现在 tools 列表中.
- 当 `effectiveAvailableSubAgents` 非空时,`spawn_sub_agent` 才可出现在 tools 列表中,且其 `agent_id` 可选范围严格来自 `effectiveAvailableSubAgents`.

说明: 本小节是“与现状不同的行为要求”,用于提醒实现者这是一次行为收敛/变更,不是文档对现状的描述.

### 4.1 `availableSubAgents` 的语义

对某个子代理 `agent`:

- 若 `agent.availableSubAgents` 为 `undefined` 或 `[]`,都视为“没有任何可 spawn 的子代理”,因此该子代理不应获得 `spawn_sub_agent`.
- 若 `agent.availableSubAgents` 为非空数组,表示“允许 spawn 数组中的子代理”(仍需按 4.2/4.2.1 做无效 id 清理与排除自身).

### 4.2 保存时清理无效 id

当用户在 UI 中保存子代理配置时:

- `availableSubAgents` 中所有不存在于当前可选子代理列表的 id,必须被自动清理.
- 清理后若结果为空数组,则该子代理等价于“没有任何可调用子代理”.

### 4.2.1 运行态兜底清理(必须)

除了 UI 保存时清理外,运行时在真正构建请求前也必须做同样的兜底清理,原因:

- 配置文件可能来自历史版本残留.
- 配置文件可能被用户手工编辑.
- 子代理可能在无 UI 场景下被执行.

因此,运行态最终使用的 `effectiveAvailableSubAgents` 必须以“清理后的结果”为准.

说明: 这样可避免由于拼写错误/历史残留导致运行时行为不可预期.

### 4.3 构建子代理请求时的 tools 注入规则

在为某个子代理构建模型请求时:

- `send_message_to_agent` 和 `query_agents_status` 是自带协作工具,保持全局可见,不随 `availableSubAgents` 开关.
- 仅 `spawn_sub_agent` 需要根据 `availableSubAgents` 决定是否出现在 `tools` 中.

特别说明(对应你反馈的现象):

- 当前实现中,`source/utils/execution/subAgentExecutor.ts` 在构建 spawn 列表时使用了 `hasSpawnWhitelist = Array.isArray(allowedSubAgentIds) && allowedSubAgentIds.length > 0` 这类判断.其结果是: 当你配置 `availableSubAgents: []` 时,代码会把它当作“没有白名单限制”,从而把所有子代理都拼进 `spawn_sub_agent` 的 description(你反馈的请求体现象).
- 本需求明确要求修复该点,并以最终运行态的 `effectiveAvailableSubAgents` 为唯一真值:
  - 当 `effectiveAvailableSubAgents` 为空(包含 `availableSubAgents: []`)时,`spawn_sub_agent` 不应出现在 tools 中.
  - 当 `effectiveAvailableSubAgents` 非空时,`spawn_sub_agent` 的 description 中展示的可 spawn 子代理列表,必须严格等于 `effectiveAvailableSubAgents`(并排除自身),不得出现配置之外的子代理.

规则(建议实现按以下顺序判定):

1. 计算有效白名单 `effectiveAvailableSubAgents`:
   - 基于 `agent.availableSubAgents`.
   - 先做“无效 id 清理”(同 4.2 的规则).
   - 再排除自身 id.
2. 若 `effectiveAvailableSubAgents` 为空(包含原始 `undefined`/`[]`/清理后为空),则 `spawn_sub_agent` **不应** 出现在该子代理请求的 `tools` 中.
3. 若 `effectiveAvailableSubAgents` 非空,则:
   - `spawn_sub_agent` 是否加入 `tools` 仍需满足运行时限制(例如最大 spawn 深度限制).若运行时不允许 spawn,则不加入 `spawn_sub_agent`.
4. `spawn_sub_agent` 工具的参数 `agent_id` 的可选范围,必须来自 `effectiveAvailableSubAgents`.
5. `spawn_sub_agent` 的 description 中展示的“Available agents you can spawn”列表,也必须来自 `effectiveAvailableSubAgents`(并排除自身),避免提示与实际可选范围不一致.
6. 执行侧兜底校验(必须): 即使由于历史 bug/模型幻觉/外部注入等原因,`spawn_sub_agent` 被调用了,执行器也必须再次按 `effectiveAvailableSubAgents` 做校验:
   - 若 `effectiveAvailableSubAgents` 为空,必须拒绝执行该 tool call.
   - 若 `effectiveAvailableSubAgents` 非空,但 `agent_id` 不在其中,必须拒绝执行该 tool call.
7. 本需求仅约束 `spawn_sub_agent` 的注入,参数范围,description 组装,以及执行侧兜底校验,不要求改变 `send_message_to_agent/query_agents_status` 的内部逻辑或可见范围.

### 4.4 与“排除自身”的关系

无论 `availableSubAgents` 如何配置,当前子代理都不应被允许 `spawn` 自己.

- UI 编辑态: 子代理列表中应排除自身,避免用户误选.
- 运行态: 即使白名单中出现了自身 id(例如旧配置残留),也必须被视为无效并被清理/忽略.

## 5. 举例覆盖需求与边缘情况

下面例子均以“为某个子代理构建一次模型请求”为场景,重点观察最终下发给模型的 `tools` 列表中 `spawn_sub_agent` 是否出现,以及其 `agent_id` 范围是否正确.另外,`send_message_to_agent/query_agents_status` 应始终存在(全局可见).

### 例 1: 字段不存在(undefined)

- 配置: `availableSubAgents` 未配置.
- 预期:
  - `tools` 仍包含 `send_message_to_agent` 和 `query_agents_status`.
  - 构建请求时 `tools` 不包含 `spawn_sub_agent`.

### 例 2: 显式空数组([])

- 配置: `availableSubAgents: []`.
- 预期: 同例 1.

### 例 3: 正常白名单

- 配置: `availableSubAgents: ["agent_explore","agent_reviewer"]`.
- 预期:
  - 构建请求时 `tools` 包含 `send_message_to_agent` 和 `query_agents_status`.
  - 若运行时允许 spawn(例如未达到最大 spawn 深度),则 `tools` 也包含 `spawn_sub_agent`.
  - `spawn_sub_agent.agent_id` 仅允许 `agent_explore,agent_reviewer`.

### 例 4: 白名单包含无效 id

- 配置: `availableSubAgents: ["agent_explore","not_exist"]`.
- 保存时: 自动清理为 `["agent_explore"]`.
- 预期: 按清理后的结果执行.

### 例 5: 白名单只包含自身

- 配置(历史残留): `availableSubAgents: ["<self>"]`.
- 保存/运行时清理后: 为空.
- 预期:
  - `tools` 仍包含 `send_message_to_agent` 和 `query_agents_status`.
  - `tools` 不包含 `spawn_sub_agent`.

### 例 6: 白名单清理后为空

- 配置: `availableSubAgents: ["not_exist_1","not_exist_2"]`.
- 保存时: 自动清理为 `[]`.
- 预期: 等价于例 2,不注入 `spawn_sub_agent`.

### 例 7: 白名单存在,但运行时不可 spawn

- 配置: `availableSubAgents: ["agent_general"]`.
- 运行时: 已达到最大 spawn 深度(或其他等价的 spawn 禁止条件).
- 预期:
  - `tools` 仍包含 `send_message_to_agent` 和 `query_agents_status`.
  - `tools` 不包含 `spawn_sub_agent`(即使白名单非空,也要受深度门禁影响).

### 例 8: 白名单非空,但运行态过滤后为空

- 配置(手工编辑或历史残留): `availableSubAgents: ["<self>","not_exist"]`.
- 运行态兜底清理:
  - `<self>` 因为“排除自身”被移除.
  - `not_exist` 因为“无效 id 清理”被移除.
  - 清理后 `effectiveAvailableSubAgents = []`.
- 预期: 等价于例 2,不注入 `spawn_sub_agent`.

## 6. 验收标准

- `send_message_to_agent` 和 `query_agents_status` 始终出现在子代理请求的 `tools` 中(全局可见).
- 当子代理 `availableSubAgents` 为 `undefined` 或 `[]` 或清理后为空时,该子代理请求的 `tools` 中不出现 `spawn_sub_agent`.
- 当子代理 `availableSubAgents` 为非空且清理后仍非空时:
  - 若运行时允许 spawn(例如未达到最大 spawn 深度),`spawn_sub_agent` 出现在 tools 中.
  - `spawn_sub_agent` 的可选 `agent_id` 列表严格受 `effectiveAvailableSubAgents` 约束.
  - `spawn_sub_agent` 的 description 中展示的可 spawn 子代理列表,必须与 `effectiveAvailableSubAgents` 一致(并排除自身),不得出现无效 id 或自身.
- UI 保存时会清理无效 id,不会把不存在的子代理 id 落盘到配置文件中.
