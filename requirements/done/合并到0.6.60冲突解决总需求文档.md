# 需求文档

## 1. 项目现状与核心目标

当前正在将 upstream/main 的变更手动合并到本地分支,本轮对应版本为 0.6.60. 这次不是只处理少量孤立冲突,而是同时涉及 API 适配器,会话编排,命令体系,UI 状态展示,i18n 文案,以及工具执行链路的成组冲突.

本需求文档的目标不是记录合并过程,而是明确本轮手动合并的最终口径,让后续处理冲突时有统一依据,避免同一类问题在不同文件里出现不一致取舍.

对非专业人士可以这样理解:

- 上游带来了一些新能力,其中一部分可以吸收.
- 但上游也带回了本地已经明确废弃的旧模式体系.
- 本轮合并要做的是: 保留本地已经确立的新架构,吸收确实有价值且兼容的上游增强,明确拒绝已经废弃的旧模式回流.

旧文档仍保留作为参考:

- `requirements/API渠道冲突合并需求文档.md`
- `requirements/合并到0.6.32冲突解决需求文档.md`
- `requirements/合并到0.6.33冲突解决需求文档.md`

但本轮总方案以本文件为准.

## 3. 已确认的全局硬约束

### 3.1 旧模式一律不回流

凡是上游重新带回来的旧模式痕迹,本轮都视为已废弃能力,不得保留其字段,文案,命令入口,状态传递,渲染逻辑,本地持久化或分支逻辑.

典型包括:

- `planMode`
- `vulnerabilityHuntingMode`
- `toolSearchDisabled` 作为旧模式配套状态时的回流用法
- `--yolo` 相关旧启动和独立模式入口
- `role` 旧体系相关入口

补充说明:

- Tool Search 这个能力本身可以保留.
- 但如果保留,必须与旧模式彻底解绑,不能再通过 `planMode`, `vulnerabilityHuntingMode`, `toolSearchDisabled` 这类旧模式状态来驱动 system prompt,命令切换,状态栏文案或页面展示.
- 本轮禁止的是“旧模式耦合形式回流”,不是默认要求删除整个 Tool Search 能力.

### 3.2 API 组遵循既有统一原则

`source/api/anthropic.ts`, `source/api/chat.ts`, `source/api/gemini.ts`, `source/api/responses.ts` 的合并,必须延续已有 API 冲突文档中已经确认的原则:

- 吸收上游多 system prompt 能力.
- 保留本地主代理/子代理 system prompt 隔离.
- 保留特殊 user 消息后插能力.
- 保留本地多模态兼容增强,尤其是 Response API 的特殊兼容路径.
- 四个适配器保持一致的参数语义与优先级模型.

### 3.3 beforeToolCall 的 exitCode = 1 采用硬阻断

`beforeToolCall` hook 返回 `exitCode = 1` 时,最终语义为:

- 不再继续执行原工具.
- 直接将 hook 输出返回给模型或界面.
- 该语义需要在 `source/utils/execution/toolExecutor.ts`, `source/utils/execution/mcpToolsManager.ts` 以及所有实际触发 `beforeToolCall` 的工具执行入口中保持一致.

这代表本轮采用更严格的安全门禁口径,不能一处阻断,另一处继续执行.

## 5. 分组手动合并原则

### 5.1 API 适配器系统提示词组

**必须保留本地**:

- 主代理与子代理的 system prompt 严格隔离.
- 统一优先级模型: 子代理自定义 system prompt -> 子代理角色定义 prompt -> 主代理自定义 system prompt -> 主代理角色定义 prompt.
- 特殊 user 消息动态插入到上下文靠后位置的能力.
- 本地多模态兼容逻辑.
- Response API 对工具多模态返回的特殊处理路径.

**必须吸收上游**:

- 多条自定义 system prompt 的数组能力.
- 四个适配器对多 system prompt 的统一支持.
- 与特殊中转渠道兼容有关的增强语义.

**明确排除项**:

- 在 options 或调用链中恢复 `planMode`, `vulnerabilityHuntingMode`, `toolSearchDisabled` 作为旧模式参数入口.
- 通过 `getSystemPromptForMode(...)` 恢复旧模式 system prompt 主路径.
- 把旧模式提示词重新注入为 system, user 或 environment_context 内容.

### 5.2 会话初始化与对话执行组

**必须保留本地**:

- `mainAgentManager` 作为当前主代理状态与 system prompt 的统一来源.
- 会话初始化继续使用当前主代理配置与自定义 system prompt 组装 system 消息.
- 已有的本地执行增强,如运行中子代理定向,回滚通知,TODO 联动等,只要不依赖旧模式就应保留.

**必须吸收上游**:

- 与稳定性,上下文展示,执行协作相关,且不依赖旧模式的通用增强.

**明确排除项**:

- 在 `initializeConversationSession` 等公共入口恢复旧模式参数签名.
- 在会话层保存,传递,消费旧模式状态.
- 让旧模式文案重新成为 system prompt 或 UI 状态来源.

### 5.3 命令体系与命令面板组

**必须保留本地**:

- 当前主代理体系下已经存在的命令集与入口.
- 与权限,主代理切换,现有面板交互一致的命令结构.
- 若保留 Tool Search 能力,它必须以独立能力存在,不能依赖旧模式开关语义.

**必须吸收上游**:

- 仅吸收和当前命令体系兼容的通用增强,不能借机恢复旧模式命令.

**明确排除项**:

- 通过旧模式命令语义重新暴露 Tool Search.
- 注册 `vulnerability-hunting` 等旧模式独立命令.
- 回流 `togglePlan`, `toggleVulnerabilityHunting`, `toggleToolSearch` 等旧模式 action 类型.
- 在命令面板中出现旧模式切换项.

### 5.4 UI 状态展示与页面组

**必须保留本地**:

- 当前主代理名称,当前配置状态,现有 YOLO 相关展示口径.
- 以主代理体系字段驱动的页面 state 和 props.
- 本地已有的配置页字段与保存语义,包括已存在的特殊配置项.

**必须吸收上游**:

- 不依赖旧模式的页面结构优化和通用交互增强.
- 上游对配置页拆分结构中可兼容的组织方式,前提是不丢本地字段与行为.

**明确排除项**:

- 在 `ChatScreen`, `ChatFooter`, `StatusLine`, `ConfigScreen` 中恢复旧模式字段.
- 旧模式状态进入页面展示,交互或持久化.
- 用旧模式状态替换当前主代理展示语义.

### 5.5 i18n 文案与类型组

**必须保留本地**:

- 与当前 UI 真实消费字段一致的类型定义.
- 与主代理体系,当前状态栏和当前页面一致的文案 key.

**必须吸收上游**:

- 与旧模式无关的通用翻译修正,如确有必要可吸收.

**明确排除项**:

- `planModeActive`, `vulnerabilityHuntingModeActive`, `toolSearchEnabled` 等旧模式文案 key.
- 为废弃旧模式补齐任何新的多语言文案.

### 5.6 工具执行与 hook 语义组

**必须保留本地**:

- `skipToolHooks` 这样的执行上下文门禁.
- 现有执行链路分层,避免把不同职责混在一起.

**必须吸收上游**:

- 对 array/object 参数的 JSON 解析增强.
- `beforeToolCall` 的统一阻断语义: `exitCode = 1` 时直接阻断原工具并返回 hook 输出.

**明确排除项**:

- 一处把 `exitCode = 1` 当阻断,另一处只当告警.
- 因吸收上游增强而删除本地执行上下文控制入口.

补充范围说明:

- 凡是仍透传旧模式字段,旧模式状态,旧模式文案 key 或旧模式命令语义的调用链文件,均属于本轮范围.
- 本轮验收不只是清理 conflict marker,还要求旧模式字段不再在类型,调用链,命令注册,UI 文案和页面状态中残留不一致.

一致性要求如下:

1. 代码层,UI 层,命令层,文案层不能对旧模式保留不同口径.
2. 任何已被排除的旧模式字段,不能只从 UI 删掉却仍留在类型或命令中.
3. 四个 API 适配器必须使用同一套 system prompt 优先级规则.
4. `beforeToolCall` 的 `exitCode = 1` 语义不能在 `toolExecutor.ts`, `mcpToolsManager.ts` 和其他实际执行入口中表现不同.
5. 配置页结构若吸收上游拆分方式,也必须完整保留本地已有字段与保存行为.
6. 若保留 Tool Search 能力,必须确认它已与旧模式 system prompt,旧模式状态栏文案和旧模式 action 彻底解绑.

## 7. 人工复查重点与风险点

### 7.1 重点风险

- 风险 1: 旧模式字段虽然从界面消失,但仍残留在类型,命令注册或状态传递中,后续又被重新接回 UI.
- 风险 2: 四个 API 适配器只改了其中一两个,导致 system prompt 行为漂移.
- 风险 3: `beforeToolCall` 在 `toolExecutor` 与 `mcpToolsManager` 中语义不一致,出现有时拦截,有时继续执行的问题.
- 风险 4: 配置页吸收上游拆分后,本地已有字段显示正常但保存链路丢失.
- 风险 5: i18n 类型和多语言内容不同步,导致构建报错或运行时缺文案.
- 风险 6: Tool Search 虽然被保留,但底层仍偷偷依赖旧模式状态或旧模式文案.

### 7.2 人工复查清单

建议人工按以下顺序复查:

1. 先看 API 四文件是否仍出现旧模式字段或旧模式提示词注入.
2. 再看会话初始化是否仍由 `mainAgentManager` 和当前 system prompt 体系驱动.
3. 再看 `useMessageProcessing.ts` 与 `chatLogic/types.ts` 是否还残留旧模式字段透传.
4. 再看命令,页面和 Tool Search 入口是否已经与旧模式解绑.
5. 再看状态栏和 i18n 是否还残留旧模式文案 key.
6. 最后看 `toolExecutor.ts` 与 `mcpToolsManager.ts` 是否统一采用硬阻断语义.

## 8. 举例覆盖需求和边缘情况

**例 1: API 适配器中的正确合并结果**

- 场景: 子代理调用 Chat 渠道,且配置了多条自定义 system prompt.
- 期望: 优先使用子代理的多条自定义 system prompt.
- 不应出现: 因为上游代码回流,又额外拼入 `planMode` 对应的默认提示词.

**例 2: 会话初始化中的正确合并结果**

- 场景: 用户打开新会话.
- 期望: 会话初始化从当前主代理配置和自定义 system prompt 生成 system 消息.
- 不应出现: 还需要从 `planMode` 或 `vulnerabilityHuntingMode` 参数决定 system 消息内容.

**例 3: 命令体系中的正确合并结果**

- 场景: 用户打开命令面板.
- 期望: 看到当前主代理体系下的命令入口.
- 不应出现: `toolsearch` 或 `vulnerability-hunting` 这类旧模式命令重新出现.

**例 4: 状态栏中的正确合并结果**

- 场景: 状态栏需要显示当前状态.
- 期望: 展示当前主代理名称,当前配置或现有状态字段.
- 不应出现: `plan mode active` 或 `vulnerability hunting mode active` 之类旧模式状态文案.

**例 5: hook 阻断语义的正确结果**

- 场景: `beforeToolCall` hook 返回 `exitCode = 1`.
- 期望: 原工具不再执行,用户或模型只收到 hook 返回的信息.
- 不应出现: 一边提示已阻断,另一边工具其实已经继续执行.

**例 6: 配置页结构吸收上游优化时的边缘情况**

- 场景: 配置页采用了上游新的拆分结构.
- 期望: 本地已有字段仍然能展示,编辑,保存,重新进入后仍保持原值.
- 不应出现: 页面看起来更整洁了,但本地新增字段被隐藏或无法保存.

**例 7: 主代理调用 API 渠道时的边缘情况**

- 场景: 主代理调用某个 API 渠道,且配置了多条自定义 system prompt.
- 期望: 只使用主代理自己的多条自定义 system prompt,不混入旧模式默认提示词.
- 不应出现: 主代理明明没有启用任何旧模式,却因为上游回流逻辑被额外注入 `planMode` 或 `vulnerabilityHuntingMode` 对应内容.

**例 8: 保留 Tool Search 但与旧模式解绑的边缘情况**

- 场景: 项目继续保留 Tool Search 能力.
- 期望: Tool Search 作为独立能力存在,不依赖旧模式状态栏文案,旧模式 action 或旧模式 system prompt.
- 不应出现: 名义上保留的是 Tool Search,实际仍要通过 `toolSearchDisabled` 或旧模式切换逻辑来驱动.

**例 9: 调用链清理不完整时的边缘情况**

- 场景: 冲突标记已经删干净,但类型文件或消息处理文件里仍透传旧模式字段.
- 期望: 类型定义,调用参数,页面 props 和命令 action 全部同步清理,保持一致.
- 不应出现: 表面 build 只剩少量问题,但后续又因为残留字段重新把旧模式接回 UI 或执行链路.
